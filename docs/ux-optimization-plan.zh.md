# UX 优化方案（按优先级）

> 基于 2026-07-02 代码走查，视角：一个第一次使用产品的普通用户。
> 现状用户路径：首页上传 MP3 / Suno 链接 → 登录弹窗 → 上传 → 跳转 /create（裁剪 + 选角色）→ 点生成并等待 → 进入工作台（歌词 → 故事 → 场景 → 图片 → 视频 → 导出）。

---

## P0 —— 直接影响信任和转化，建议先做

### 1. 首页上传的「✓ 完成」瞬间缺失

**现状**：/create 页其实已经有「✓ Song uploaded」的打勾卡片（`audio-upload-trim.tsx` 1211 行起的完成态分支）。缺的是**首页这一端**：`lyric-video-home-tool.tsx` 把组件写死为 `completionState="idle"`，上传到 100% 后没有任何成功态，直接被转圈遮罩盖住然后跳页。另外跳转后如果音频时长读取慢，用户先看到的是「Reading audio」的加载态（`shouldDeferInitialAudio` 分支），打勾出现得更晚。

**用户感受**：进度条刚满页面就变了，「传好了」这个确认瞬间被跳过；到了新页面还要再等一次加载。

**方案**：首页进度条到 100% 时先切换为绿色 ✓「上传完成」态，停留约 800ms 再跳转；跳转时把时长等元数据一起带过去（首页裁剪阶段已经解析过时长），让 /create 直接命中打勾卡片而不是先进「Reading audio」。

**改动点**：`lyric-video-home-tool.tsx`（去掉 `completionState="idle"`、加延时）、`writeHomeUploadedAudio` 确保 durationSeconds 始终写入。工作量：小。

### 2. 上传失败也照样跳转到 /create

**现状**：`uploadAndContinue` 的 catch 分支只弹一个 toast，`finally` 里仍然执行跳转（代码注释："still send the user to the creator"）。用户带着一个空状态落在创作页，歌没了，得重新选文件。

**用户感受**：「报了个错还把我带走了，我的歌呢？」

**方案**：失败时留在首页原地，进度条区域变为红色错误态 + 「重试上传」按钮（文件对象还在内存里，重试不需要重新选择）。连续失败 2 次以上再提示检查网络/换文件。

**改动点**：`uploadAndContinue` 把跳转移出 `finally`；`audio-upload-trim.tsx` 加失败/重试态。工作量：小。

### 3. 风格是盲选——没有基于用户歌曲的视觉参考

**现状**：`customize-panel.tsx` 的风格选择只有 9 张通用静态示意图（`STYLE_OPTIONS.previewSrc`），不是用户这首歌的效果。流程上必须先确认故事、锁定风格，才能生成场景和图片；而 `deriveStoryDirectionLockState` 规定场景一旦创建，故事和风格永久锁定。也就是说：用户在看到任何一张自己歌曲的成图之前，就要做出不可逆的风格决定。

**用户感受**：「我怎么知道 anime 风格配我这首歌好不好看？换一个试试？——哦，锁了。」

**方案（分两步落地）**：

- **第一步（核心）**：故事确认后不直接生成全部场景图，而是自动挑 3 个代表性场景（开头 / 高潮 / 结尾各一）先生成 3 张预览图，展示在风格选择区上方。用户切换风格时，用新风格重新生成这 3 张（只花 3 张图的成本），满意后点「确认风格，生成全部场景」。
- **第二步（增强）**：3 张预览图按风格缓存，用户在几个风格间来回切换时直接读缓存，不重复扣费。

**改动点**：`customize-panel.tsx`（预览区 UI）、生成 API 支持「仅指定场景」的部分生成（scenes-panel 已有按选中场景生成的能力，可复用）、credits 计费按 3 张计。工作量：大，但这是产品的核心决策点，值得投入。

### 4. 风格/故事锁死后没有出口

**现状**：锁定提示是「Direction settings are locked after scenes have been created」——只告诉用户不行，不告诉用户怎么办。

**方案**：锁定态下提供「重新开始方向设计」按钮，明确告知代价：「将清除已生成的 N 个场景和图片，重新生成约需 X credits」。让用户自己权衡，而不是死路一条。文案同时改成中性描述（见 #10）。

**改动点**：`customize-panel.tsx`、`field-block.tsx`（锁定态 UI），后端加一个重置场景的接口。工作量：中。

## P1 —— 流程顺滑度

### 5. 首页 → /create 的过渡衔接

**现状**：/create 已经正确显示「✓ Song uploaded」卡片 + 裁剪 + 人物选择，这部分没问题。要打磨的是过渡本身：首页遮罩文案「Opening the creator...」和 /create 可能出现的「Reading audio」加载态，让两页之间有一段视觉断档。

**方案**：与 #1 合并处理——首页展示成功态后再跳转、元数据随行，/create 首屏直接是打勾卡片，过渡就连贯了。

**改动点**：随 #1。工作量：并入 #1。

### 6. 登录弹窗：邮箱表单在上，Google 在下

**现状**：`lyric-video-home-tool.tsx` 的登录弹窗里，邮箱+密码表单占主位，Google/GitHub 按钮在下方。

**方案**：新用户绝大多数走一键社交登录。把 Google 大按钮放最上面，邮箱表单收进「用邮箱登录」折叠项。注意：社交登录 `callbackURL: "/create"` 会整页跳转，需确认 pendingUpload 的文件在跳转后还能续传（内存中的 File 对象会丢失）——如果不能，提示用户「登录后请重新选择文件」比静默丢失好。

**改动点**：弹窗布局调整；验证社交登录回调路径。工作量：小（布局）+ 中（回调续传验证）。

### 7. 点「生成」后的长时间黑盒等待

**现状**：/create 点生成后，`use-lyric-video-creation-flow.ts` 用 `wait: true` 同步等待整个歌词转写 + 故事生成完成，期间只有一个笼统的 spinner。这个请求可能跑几十秒，还有 Vercel 函数超时风险（AGENTS.md 自己就要求 API 尽快返回、轮询要有明确 owner）。用户中途刷新或离开，状态就丢了。

**方案**：生成改为立即返回 + 跳转进工作台，工作台里用 RoadmapGuide（已存在）展示「正在识别歌词… → 正在构思故事…」的分步进度。这样用户等待时已经身处工作台，可以先看界面熟悉功能，跳出率会低很多。

**改动点**：`generate` API 支持异步模式（代码里已有 `stopAfter`/`pipelineStage` 等基础）、前端轮询接管。工作量：大，但同时解决了架构隐患。

### 8. 角色选择也是盲选

**现状**：/create 页的 `CharacterPresetPicker` 在任何成图之前出现，且不知道最终会套什么风格。

**方案**：短期先给每个角色预设加缩略图（如果还没有）；长期并入 #3 的预览流程——3 张预览图里就包含所选角色，一并确认。工作量：随 #3 走。

## P2 —— 细节打磨

### 9. 「Keep this page open while the audio uploads」

依赖用户开着页面才能完成关键步骤。上传阶段这是合理限制，但文案可以更安心一些：「正在上传（34%）…请稍候」。生成阶段的页面依赖由 #7 解决。

### 10. 锁定/报错文案偏惩罚性

「locked after scenes have been created」「Story confirmed. Click Edit Story below to make changes.」这类文案是从系统角度写的。统一改成「为保证画面一致性，风格在场景生成后保持固定。想换风格？→ 重新设计方向」的引导式写法。顺带全站过一遍 toast 文案。

### 11. 成本透明度

场景面板里有单步的 credits 显示（「Generate N Scene Images (X credits)」），但用户在 /create 点生成前，不知道走完全程大概要花多少。建议在生成按钮附近给一个总成本区间预估：「预计全程消耗 80–150 credits（取决于场景数）」。

### 12. 风格静态示意图与实际产出的差距管理

9 张 `previewSrc` 示意图质量决定了用户预期。建议这 9 张图用同一个场景/同一个人物在 9 种风格下的产出（而不是 9 张不同内容的图），用户对比的才是「风格」本身。

## 建议实施顺序

| 顺序 | 项目 | 工作量 | 理由 |
|---|---|---|---|
| 1 | #1 + #2 上传成功/失败反馈 | 小 | 一次改动同一个组件，立刻消除最刺眼的粗糙感 |
| 2 | #6 登录弹窗布局 | 小 | 注册转化入口 |
| 3 | #4 锁定出口 + #10 文案 | 中 | 消除死路 |
| 4 | #3 三张预览图选风格 | 大 | 核心体验升级，做完产品质感上一个台阶 |
| 5 | #7 生成异步化 | 大 | 体验 + 架构双收益，可与 #3 排期协调 |
